## Critical（動線が止まる / データ消失）

### C-1: ポータルのオンボーディングが全クリックをブロックし、操作パネルが画面外に見切れる
- 症状: 初回ログイン直後、`fixed inset-0 z-[100]` のオーバーレイが全面を覆い、説明パネルが**ビューポート下端より下に描画**されて「次へ/スキップ」が押せない。唯一の脱出口はパネル右上の小さな×のみ（それも見切れる高さでは不可）。スポットライトの白抜きが対象カードの中身（承認/修正依頼ボタン）を隠す。
- 根本原因: `PortalOnboardingWalkthrough.tsx:157` パネル位置 `top: targetRect.bottom + 16` に**ビューポートへのクランプがない**。ターゲットが画面下部にあるとパネルがはみ出す。
- ユーザーフィードバック（要対応）: 「実際のボタンにフォーカスして説明してほしい。モーダル表示だけでは古い」— spotlight 実装(`useSpotlightRect`)は存在するが、①パネル位置が壊れる ②ハイライト対象自体もクリック不可 ③ステップ文言が「黄色いバッジ」と**言葉で**説明しており対象を指していない。
- 対応方針: パネル位置のビューポートクランプ＋ターゲットの実座標にパネルを吸着（上下自動反転）＋ハイライト対象は pointer-events を許可し「実際に押して次へ進む」インタラクティブツアー化＋Esc/背景クリックで常に脱出可能に。内部版 `InternalOnboardingWalkthrough.tsx` も同実装のため同時修正。

### C-2: クライアントの「リクエストを送る」が 500 で消失
- 症状: 機能要望を送信→無反応→送信リクエスト一覧 0 件。クライアントの入力内容が完全に失われる。
- 根本原因: `POST /api/portal/requests`（`src/app/api/portal/requests/route.ts:240-254`）の tasks INSERT に `created_by` が無い → `23502 null value in column "created_by" violates not-null constraint`。
- 対応方針: `created_by: user.id` を追加＋**再現テスト**（TDD: 送信→201→一覧反映）＋失敗時にフォーム値を保持したエラー表示（現状の挙動も要確認）。

### C-3: ポータル「承認履歴」が常に空（操作証跡が見えない）
- 症状: 修正依頼直後でも「アクション履歴はまだありません」。クライアントは自分の操作が届いたか確認できない。
- 根本原因: サーバログ `[Portal History] audit query error: PGRST200` — `audit_logs` と `tasks` の間に FK リレーションが無く、埋め込み JOIN クエリが失敗。エラーは握りつぶされて空状態文言を表示。
- 対応方針: JOIN を2段クエリ（audit_logs 取得→task_id で IN 検索）に変更 or FK 追加のマイグレーション。**クエリ失敗時は「読み込みに失敗しました」を出す**（空状態と失敗の混同は信頼を損なう）。

## High（気づけない / 信頼を損なう）

### H-1: クライアントの修正依頼が内部のアプリ内通知に届かない
- 修正依頼は Slack 通知のみ発火（`POST /api/slack/notify` 200 を確認）。**受信トレイには何も来ない**（既存通知は2/14のシードのみ）。Slack未連携の組織では完全に無音。
- さらにタスク側: ボールは社内に戻るが「修正依頼あり」の状態表示がなく、コメントは折りたたまれたコメント欄の中。担当者は**未設定にクリア**され（⚠社内メンバーに変更推奨とだけ表示）、誰も拾わないリスク。
- 対応方針: notifications テーブルへの in-app 通知作成（種別: 修正依頼）＋タスク行/インスペクターに「修正依頼」バッジ＋担当者は元の内部担当（作成者）へ自動で戻す。

### H-2: ポータルダッシュボードの進捗情報が自己矛盾
- 「現在のステータス: 順調に進行中／全タスクが予定通りに進行中です」の同一画面に「完了予定: 2/28 **127日超過**」「次回納品予定 2025/10/15 **(263日超過)**」。期限超過をリスク判定が拾っていない（or 判定対象が別データ）。クライアントに最も見える画面での矛盾は信頼を直撃する。
- 対応方針: リスク判定に期限超過（完了予定・次回納品）を反映。超過があるなら「遅延あり」表示に。

### H-3: クライアントの表示名が「ゲスト」
- client1（鈴木一郎）でログインしてもサイドバー下は「ゲ ゲスト」。`PortalLeftNav.tsx:120` の `userName || userEmail?.split('@')[0] || 'ゲスト'` まで userName/userEmail が渡っていない。JWT の `user_metadata.name` には「鈴木 一郎」が存在するため取得側の配線漏れ。
- 「誰としてログインしているか」が分からないのは承認操作を行うポータルでは不安要素。

### H-4: 判断材料を見せずに「承認」させる
- ダッシュボードのカードに承認/修正依頼ボタンが直接あり、詳細パネルを開いても**タイトルと作成日時のみ**（説明・成果物リンク・依頼メッセージの表示領域がない）。何を確認して承認するのかが分からない。
- 対応方針: 詳細パネルに説明(description)・添付/成果物・依頼コメントを表示。説明が無い場合は内部側の作成UIで「クライアントに見せる説明」を促す。

### H-5: 「(263日超過)」ラベルがカードからはみ出す（ユーザー報告）
- 次回納品予定カードで日付＋超過ラベルが折り返し、カード外にはみ出して見える。長い数値でのレイアウト崩れ。`min-w-0`/`truncate`/ラベルを別行に分離で対応。

## Medium（認知負荷・ラベル表記）

### M-1: 「確認・承認」系ラベルが7種類乱立
同じ「相手の対応待ち/承認」概念に対し: **確認待ち**（内部フィルタ・バッジ）/ **外部確認待ち**（内部リスト）/ **承認待ち** / **承認確認中**（ステータス値）/ **承認済** / **要確認**（ポータル詳細バッジ）/ **要対応**（ポータルナビ）。
- 内部→「外部確認待ち」、ポータル→「要対応」と、**同じタスクの同じ状態が画面で呼び名が変わる**。
- 対応方針: 用語辞書を1枚定義（例: 内部=「クライアント確認待ち」、ポータル=「要対応」の2語に集約し、承認フロー(社内)は「社内承認」と明示的に別語にする）。UI_RULES に追記。

### M-2: タスク作成の「担当者」と「関係者・外部（必須）」の二重指定
- ボール=外部にすると「関係者・外部（必須）」が出現するが、直下の「担当者」にも同じクライアント名が並ぶ。違いが説明されておらず、どちらを選ぶべきか初見では判断不能。
- さらに必須未選択で「作成」を押しても**無反応**（エラーメッセージなし・ボタンは活性に見える）。
- 対応方針: 担当者(assignee)を選べば関係者を自動補完 or 統合。バリデーションエラーの表示。

### M-3: ステータス6値の意味が重複
「未着手」と「ToDo」の区別がつかない。「検討中」「承認確認中」も文脈依存。ボール(2値)×ステータス(6値)×承認フローの3軸は初見の作業者に過剰。最低限「未着手/ToDo」の統合を推奨。

### M-4: 「承認を依頼」がクライアント宛だと誤解される
クライアント確認タスクを開いて「承認フロー→承認を依頼」を押すと、候補は**社内メンバーのみ**（佐藤・山田）。クライアントに承認を求める動線は「ボール=外部」側で、同じインスペクター内に2つの承認概念が並ぶ。ラベルを「社内承認を依頼」にする＋ヘルプ1行で解消可能。

### M-5: 常時表示の説明文が多い（認知負荷）
- ポータル各ページ冒頭に毎回説明文（「プロジェクトの全体進捗と、あなたの確認が必要な項目です。」「過去のアクションと完了したタスクの履歴です」等）。初回以降はノイズ。
- **視点の誤り**: クライアントに見せる「送信リクエスト」ページの説明が「**クライアントから**送信されたバグ報告・機能要望・質問の一覧です。」と内部視点の文言。→「あなたが送信したリクエストの一覧です」。
- 対応方針: ページ説明はツールチップ/初回のみに。文言はすべて読者視点で統一（社内画面=社内視点、ポータル=クライアント視点）の表記ルールを UI_RULES に追加。

### M-6: 内部タスク一覧の常時表示アクションが多い
全行に「選択」「未完了に戻す」「ステータスを変更（現在: 完了）」が常時出る（hover時のみで良い）。また「テストです」「テストで」等のゴミデモデータがポータル側のタスク一覧にもそのまま見えている（デモ体験の品質低下）。

## 良かった点（維持すべき設計）
- ボール往復のコア動線はデータ層まで一貫して動く（外部確認待ち→ポータル要対応→修正依頼→ボール社内復帰→コメント記録）。
- 修正依頼にコメント必須（「※修正依頼にはコメントが必要です」）は適切なガード。
- ポータルの語彙は概ねクライアント向けに翻訳済み（ボール/社内などの内部用語が漏れていない。タスク一覧は すべて/進行中/完了 のみ）。
- タスク作成シートでボール・担当・マイルストーンを一括設定できる。
- ポータルのリクエストフォーム自体の構成（種別3択＋タイトル＋内容）はシンプルで良い。
