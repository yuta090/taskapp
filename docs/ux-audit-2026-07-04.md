# UX監査: TaskApp / AgentPM — 初回ユーザーの登録〜利用開始フロー

**日付**: 2026-07-04
**シナリオ**: 「開発会社のPMとして新規登録して使い始める」＋「招待されたクライアントとして初めてポータルを開く」
**ブラウザ**: Playwright（localhost:4000, develop ブランチ）
**ビューポート**: デスクトップ (1200px) + モバイル (375px)
**スクリーンショット**: `docs/ux-audit-assets/2026-07-04/`

## サマリー

会員登録画面そのものは見た目・モバイル対応ともに良好だが、**初回体験の根幹が2箇所で壊れている**。(1) クライアントポータルが `QueryClientProvider` 欠落で全ページクラッシュしており、招待されたクライアントは何もできない（f9ab6a0 のリグレッション）。(2) チームメンバー招待UIがスタブのままで、登録直後の「チームを作る」ステップが行き止まりになる。加えて、ブランド名が AgentPM / TaskApp で分裂しており、招待メール（TaskApp名義）から着地したページ（AgentPM表記）で初対面のクライアントに不信感を与える。

## Findings

### Critical（タスク完了を阻害）

- **クライアント/ベンダーポータルが全ページクラッシュ**: クライアントとしてログイン（またはポータル招待リンクから参加）すると「エラーが発生しました 予期しないエラーが発生しました。」のエラーバウンダリだけが表示される。初回クライアントはオンボーディングウォークスルーすら見られない。
  - *原因*: `PortalLeftNav` → `usePortalVisibilityForPortal` (`src/lib/hooks/usePortalVisibility.ts:72`) が React Query の `useQuery` を使うが、`QueryProvider` は `src/app/(internal)/layout.tsx` と `src/app/settings/layout.tsx` にしかなく、`src/app/portal/` / `src/app/vendor-portal/` 配下には無い。`PortalShell` を使う全ページ（dashboard / tasks / all-tasks / requests / scheduling / wiki / settings / task/[taskId] / vendor-portal）が巻き込まれる。
  - *混入コミット*: f9ab6a0「feat: ポータル表示制御機能」
  - *Where*: `/portal` 全域, `/vendor-portal` 全域
  - *Screenshot*: `portal-first-run.png`
  - *Fix*: `src/app/portal/layout.tsx`（および vendor-portal）に `QueryProvider` を追加。再発防止として、ポータルをデモクライアントで開くだけのE2Eスモークテスト（`page.on('pageerror')` 監視）を `tests/e2e/` に追加する（TDD: まず落ちるテストを書く）。

- **メンバー招待がスタブのまま行き止まり**: 設定→メンバーで招待を実行すると「〜への招待メール送信機能は準備中です。直接メンバーを追加する場合は、まずユーザーにアカウント作成を依頼してください。」と表示される。登録→チーム招待という初回の主要動線が完了できない。しかも招待メール基盤（`/api/invites` + `sendInviteEmail`）は実装済みで、UIだけが未接続。
  - *Where*: `src/app/settings/members/page.tsx:179`
  - *Fix*: 既存の `POST /api/invites` に接続する。メールアドレス＋ロール（メンバー/クライアント）を渡すだけで動く状態になっている。

### High（混乱・大きな摩擦）

- **ブランド名の分裂（AgentPM vs TaskApp）**: ページタイトル・ロゴは「AgentPM - つくることに、集中できる」、一方で招待メール件名は「【TaskApp】〜」、ポータルバッジ・環境変数 `NEXT_PUBLIC_APP_NAME` は TaskApp。**「TaskApkからの招待メール→AgentPMという別名のサイトに着地」は、初対面のクライアントにはフィッシングに見える**。
  - *Where*: `src/app/layout.tsx`（タイトル）、`src/lib/email/index.ts`、`src/components/auth` ロゴ、マニュアル `docs/manual/`
  - *Fix*: 名称をどちらかに統一（`NEXT_PUBLIC_APP_NAME` を単一ソースにし、メール/ロゴ/タイトル/マニュアルすべて参照させる）。

- **テスト用デモアカウント7件が本番でも表示される**: ログイン画面下部の「テスト用デモアカウント」（実パスワード付きワンクリックログイン）は環境ゲートが無く常時レンダリングされる（`src/app/(auth)/login/LoginClient.tsx:10` の `DEMO_ACCOUNTS` は無条件で map）。初回の実ユーザーには「テスト中の未完成品」に見え、かつ認証情報がバンドルに露出するセキュリティ問題でもある。
  - *Fix*: `NEXT_PUBLIC_SHOW_DEMO_ACCOUNTS`（または `NODE_ENV !== 'production'`）でゲートする。

- **会員登録フローの組織作成が不安定（サイレント失敗→二度聞き）**: `/signup` は 組織名 を入力させ `rpc_create_org_with_billing` を即時呼び出すが、メール確認が有効な場合セッションが無いままRPCが実行され、失敗しても `console.error` のみで成功画面を表示する（`src/app/(auth)/signup/page.tsx:67-72`）。失敗していた場合、メール確認後に `/onboarding` で**組織名をもう一度入力させられる**（`user_metadata.org_name` に保存済みなのにプレフィルもされない）。Google登録ユーザーも毎回手入力。
  - *Fix*: 組織作成はメール確認後（`/auth/callback` またはDBトリガー）に `user_metadata.org_name` を使って一元化する。`/onboarding` では `org_name` をプレフィルする。
  - *補足（要確認）*: 現状RPCはanonでも実行できる可能性が高く（下記セキュリティ所見）、その場合「確認前に組織が作られる＝未確認ユーザーの孤児組織が溜まる」という別の問題になる。どちらに転んでも作り直しが必要。

- **登録完了画面が次の行動を誤誘導**: 「確認メールを送信しました」画面のリンクが「ログインページへ」だが、実際はメール内リンクをクリックすれば自動ログイン→組織作成/プロジェクトへ進む。先にログインを試すと「Email not confirmed」系のエラーで詰まる。迷惑メールフォルダへの言及も無い。
  - *Where*: `src/app/(auth)/signup/page.tsx:90-113`
  - *Fix*: 「メール内のリンクをクリックすると自動的にログインされます。届かない場合は迷惑メールフォルダをご確認ください」に変更し、再送ボタンを付ける。

- **クライアント向けマニュアルが実装と乖離**: `docs/manual/client/getting-started.md` は件名「プロジェクトポータルへの招待」・「アカウント作成不要／トークンベース」と説明するが、実際の件名は「【TaskApp】〈組織名〉からプロジェクトへの招待」で、ポータル参加時に**パスワード設定＝アカウント作成が必須**（`src/app/portal/[token]/page.tsx`）。マニュアルを信じたクライアントが混乱する。
  - *Fix*: マニュアルを現行フローに合わせて書き直す（doc-index-updater 案件ではなく内容の書き直し）。

### Medium（回避可能だが不親切）

- **バリデーションエラーの位置と必須マーク**: 短いパスワードのエラー「パスワードは8文字以上で入力してください」がフォーム最上部（組織名の上）に表示され、パスワード欄はハイライトされない。必須マーク（*）も無し。パスワードの表示切替（目アイコン）も無い。
  - *Where*: `src/app/(auth)/signup/page.tsx:144-148`（login/reset も同パターン）
  - *Fix*: フィールド直下エラー＋ `aria-invalid` ＋枠色変更。可能ならリアルタイム検証。

- **既登録メールで登録すると偽の成功画面（可能性大）**: Supabaseはメール確認有効時、既存メールへの `signUp` を成功風レスポンスで返す（列挙攻撃対策）。コードの「already registered」分岐はおそらく発火せず、ユーザーは「確認メールを送信しました」を見るがメールは届かない。
  - *Fix*: 成功画面に「既にアカウントをお持ちの場合はメールは届きません。ログインまたはパスワードリセットをお試しください」の一文を追加。

- **招待メールの文脈情報が薄い**: 件名・本文に「何のプロジェクトか」はあるが、**期待するアクション（何を・いつまでに確認するのか）や差出人の連絡先が無い**。差出人名は `user_metadata.full_name || email || '管理者'` で、Googleログインでないメンバーは生メールアドレスが差出人名になる。
  - *Where*: `src/lib/email/index.ts`, `src/app/api/invites/route.ts`
  - *Fix*: 招待作成時に一言メッセージ（任意入力）を添えられるようにし、差出人はプロフィール表示名を優先。承認依頼メール（`approval.ts`）は期限・タスク説明の1行要約を含める。

- **ログイン済みユーザーが /login /signup をそのまま開ける**: 認証済みでもフォームが再表示され、現在ログイン中である表示も無い。デモ切替では便利だが、実ユーザーには「ログインできていない」ように見える。
  - *Fix*: 認証済みならリダイレクト、少なくとも「〜としてログイン中」バナーを表示。

- **auth画面でコンソールエラーが常時発生**: 未ログイン状態の全auth画面で `AuthSessionMissingError: Auth session missing!` がエラーとして出力される（期待される状態なのにエラー扱い）。Nextのdevオーバーレイにも「1 Issue」が常時点灯し、E2Eの `pageerror` 監視ノイズにもなる。
  - *Fix*: セッション未存在は正常系として握りつぶす（`getUser` 前に `getSession` チェック等）。

### Low（磨き込み）

- **ログインと登録でGoogleボタンの位置が逆**: `/signup` はGoogleが最上部、`/login` はフォームの下。動線の一貫性がない。
- **オンボーディングのフラグがlocalStorageのみ**: `taskapp_internal_onboarded` / `taskapp_portal_onboarded` は端末依存。別端末・キャッシュクリアで再表示、逆にプロフィールでは追跡不可。サーバー側（profiles）に持つ方が堅い。
- **内部ウォークスルーの表示タイミング**: 大量のタスクが並ぶ画面の上にモーダルが被さる。ステップ2「ボールの概念」・ステップ3「クライアントに公開」の文言自体は良いが、対象UI（公開トグルや黄色バッジ）を指すスポットライトが無いため「どこの話か」が分からない。
- **`FROM_EMAIL` 未設定時のデフォルトが `noreply@taskapp.example.com`**: 未設定環境では送信失敗または迷惑メール行きになる。起動時に警告を出すべき。

### セキュリティ所見（UX外・要対応）

- **`rpc_create_org_with_billing` が SECURITY DEFINER で認可チェック無し**: `auth.uid()` と `p_user_id` の一致検証が無く、`20260703_009_rpc_authz_hardening.sql` でも未対応。EXECUTE権限がanonに残っていれば、未認証で任意の `p_user_id` に組織・owner権限・billingを無制限に作成できる。`rpc_accept_invite` も同パターン（こちらはトークンが秘密情報だが、`p_user_id` の縛りは無い）。
  - *Where*: `supabase/migrations/20240103_000_auth_billing.sql:131`
  - *Fix*: RLS/RPC硬化ストリーム（`security/*`）で `auth.uid()` 検証＋anonからのEXECUTE剥奪。サインアップフロー再設計（上記High）とセットで。

## What Works Well

- 登録・ログイン画面の視覚デザインとモバイル(375px)対応は良好（横スクロール無し、タッチターゲット十分）。
- スキップリンク（メインコンテンツへスキップ）などアクセシビリティ配慮がある。
- Freeプラン内容の明示（プロジェクト5件/メンバー5名/クライアント5名）は登録前の不安を減らす良いパターン。
- 内部/ポータル両方に初回ウォークスルーが存在し、「ボール」「クライアントに公開」という独自概念を先回りで説明している。文言も簡潔。
- 招待・承認メールはHTML/プレーンテキスト両対応、XSSエスケープ済み、有効期限明記、「心当たりがない場合は無視してください」の定型文もある — 骨格は良い。
- 期限超過バナー「期限超過 8件 → ガントで確認」のような、次の行動につながる通知設計。

## Recommendations（優先順）

1. **ポータルのクラッシュ修正**（QueryProvider追加＋ポータルのE2Eスモークテスト）— クライアントの初回体験はこれが直るまでゼロ点。
2. **メンバー招待UIを実装済みAPIに接続** — 登録直後の主要動線の行き止まりを解消。
3. **ブランド名の統一**（AgentPM/TaskApk → 単一ソース化）と**デモアカウントの環境ゲート** — 初対面の信頼性。
4. **サインアップ→組織作成フローの再設計**（メール確認後に一元化、`org_name` プレフィル、完了画面の文言修正、RPC認可強化とセット）。
5. **クライアント向けマニュアルの現行化**と招待メールへのひと言メッセージ欄追加。
