# モデル・オーケストレーション運用設計（Model Routing）

> 目的：Claude Code での TaskApp 開発・運用において、**どの作業をどのモデルに任せるか**を型化し、
> 「Fable を起動する前に、下位モデルで準備（収集・整形・チェック）を完璧に済ませる」運用を実現する。
>
> ステータス：**運用中**（役割分担表 合意済み）— 定義ファイル（`.claude/agents`, `.claude/workflows`, スキル手順書, CLAUDE.md 運用ルール）へ固定済み。

---

## 1. 思想（Routing Philosophy）

1. **Fable 起動前に机を完璧にする**：収集・整形・チェックなどの準備は全部下位モデルに済ませ、
   Fable には「整った状態での最終判断」だけをさせる。
2. **Fable に任せるのは 3 条件を満たす判断だけ**：`型がない × 失敗コストが大きい × 全体を見る必要がある`。
3. **モデル振り分けは原則自動**：サブエージェント定義ファイル（`model:`）と Workflow 台本（`model`/`effort`）で
   勝手に切り替わる。スキルにはモデルを書かず、モデル指定済みのエージェントを呼ぶ形に一元化する。
4. **メインセッションのモデルだけは自動化不可**：仕様上 Claude 自身では変更できない。
   ユーザーが `/model` で切り替える。Claude は「今 Fable に切り替えるべき／Opus に戻してよい」の**提案まで**を担当する。
5. **単発・未確定の業務は固定しない**：1 回やってみて、繰り返すと分かった時点で初めて定義ファイル化する。

## 2. モデル 4 段の判断基準

| モデル | 使いどころ | 判定キーワード |
|--------|-----------|----------------|
| **Fable 5** | 型がない × 失敗コスト大 × 全体俯瞰が必要な**戦略・設計・全体判断** | 新規アーキテクチャ、DBスキーマの根幹、マルチテナント境界、重大なセキュリティ設計、後戻り困難な意思決定 |
| **Opus 4.8** | 日常の**判断・レビュー・執筆** | コードレビュー、リファクタ方針、仕様執筆、UXコピー、承認/差し戻し判断、既存拡張のDDL |
| **Sonnet 5** | 型が決まった**量産・実行** | 仕様に沿った実装、LPコンポーネント実装、バグ/UXパッチ、マニュアル量産、seed |
| **Haiku 4.5** | 機械的な**変換・検査** | レポート集計、lint/test結果の解釈、デザイントークン準拠チェック、SPEC_INDEX更新、インベントリ |

> 対応の原則：業務性質 **A→Fable / B→Opus / C→Sonnet / D→Haiku**。
> （A=型なし×失敗コスト大×全体俯瞰 / B=判断あり / C=量産実行 / D=機械的）

## 3. 業務の洗い出し（下位モデルによる調査結果サマリ）

調査は Sonnet 5（意味理解を伴う洗い出し）＋ Haiku 4.5（機械的インベントリ）で実施。詳細は本ファイル §4 の表に統合。

- **プロダクト運用**：client-onboarding / meeting-flow / review-cycle / scheduling-wizard / agentpm(ボール判定) / 各種レポートコマンド（ball-status, open-reviews, pending-scheduling, activity-digest, project-overview, project-status）
- **開発・実装**：新機能設計・実装、バグ/UXパッチ、リファクタ、LP実装
- **品質保証**：lint、Vitest、Playwright E2E、コードレビュー、セキュリティレビュー、デザインシステム準拠チェック
- **DB・マイグレーション**：DDL差分作成、RLS設計・監査、seed投入、バックアップ、pg_cron、トークンクリーンアップ（未実装）
- **デザイン/LP**：プロトタイプ改訂、LPコピー改訂、LPコンポーネント実装、デザイントークン管理
- **レポート/集計**：burndown、CSV export、CLI利用統計、Stripe課金、管理パネル運用
- **ドキュメント**：SPEC作成、SPEC_INDEX更新、マニュアル整備、UX改善記録、DECISIONSログ

補足所見：`.github/` なし（CI未構築の可能性）、アプリ層の定期ジョブは `pg_cron` に限定、`docs/EMAIL_APPROVAL_TODO.md` にトークン期限切れ削除の未実装 TODO あり。

## 4. 役割分担表（作業／モデル／理由／動かし方）

> 記号：**★=繰り返し発生（定義ファイル化の候補）** / ☆=都度・単発（当面は思想ルールに留める）

### プロダクト運用

| 作業 | モデル | 理由 | 動かし方 |
|------|--------|------|----------|
| ★クライアントオンボーディング | Opus 4.8 | 手順は確立だがクライアント関係に直結する判断あり(B) | スキル `client-onboarding` → メイン（Opus） |
| ★会議フロー（作成〜議事録〜タスク化） | Opus 4.8 | 議事録からのアクション抽出に判断(B) | スキル `meeting-flow` → メイン（Opus） |
| ★レビューサイクル（承認/ブロック判断） | Opus 4.8 | 差し戻し妥当性の日常判断(B) | スキル `review-cycle` → メイン（Opus） |
| ★ボール判定（client/internal） | Opus 4.8 | 誤判定がクライアント関係に直結(B) | スキル `agentpm` → メイン（Opus） |
| ★日程調整ウィザード | Sonnet 5 | 提案作成は型化された実行(C) | スキル `scheduling-wizard` → サブエージェント（Sonnet） |
| ★レポート群（ball-status / open-reviews / pending-scheduling / activity-digest / project-overview / project-status） | Haiku 4.5 | 集計・整形の機械処理(D) | コマンド/スキル → サブエージェント（Haiku）、将来 cron+Slack 自動配信 |

## 5. 定義ファイル化の方針（合意後に実施）

- **①サブエージェント定義**（`.claude/agents/*.md` の `model:`）：★のうち反復性が高いもの
  （例：`impl-runner`=Sonnet、`report-collector`=Haiku、`design-system-checker`=Haiku、`code-reviewer`=Opus、`doc-index-updater`=Haiku、`migration-writer`=Opus）
- **②Workflow 台本**：多段・並列が必要なもの（例：レビュー＝dimensions→find→verify、LP量産、リリース前チェック）に `model`/`effort` を明記
- **③スキル手順書**：スキル自体にはモデルを書かず、①のエージェントを呼ぶ形へ一元化
- **④CLAUDE.md 運用ルール**：本ファイルの思想＋「メイン切替の提案ルール」を要約して追記
- **☆（都度・単発）**は定義ファイル化せず、④の思想ルールに従って都度判断

## 6. メインセッション切替の提案ルール（草案）

Claude は次を検出したら、作業を止めてユーザーに切替を提案する（自分では切替不可）：

- **「Fable に切り替えるべき」提案**：新規アーキテクチャ設計 / DBスキーマの根幹・RLS境界の新規設計 / 決済・認証・承認トークン等の重大セキュリティ設計 / 後戻り困難な全体意思決定 に着手する時。
  → 提案時は「準備（収集・整形・チェック）が下位モデルで完了していること」を先に示す。
- **「Opus に戻してよい」提案**：上記の重い判断が終わり、以降が実装・レビュー・執筆などの日常作業に移る時。
- 提案フォーマット例：`⏸ ここは「型なし×失敗コスト大×全体俯瞰」に該当します。/model で Fable に切り替えてください（机は整っています：〇〇済み）。`
