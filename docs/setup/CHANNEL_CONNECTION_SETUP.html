AI秘書 チャネル接続セットアップガイド

各チャットサービスと AI秘書 をつなぐために、運用者（あなた）が設定する箇所を1枚にまとめた手順書です。
受信（メッセージの自動取り込み）を有効にするサービスごとに、①発行する鍵 ②管理画面に登録する値 ③サービス側の Webhook 設定 の3点を示します。

用語：本書の <アプリのドメイン> は TaskApp を公開しているURL（例 https://app.example.com）、 {accountId} は管理画面でチャネル資格情報を登録した後に発行されるアカウントIDに読み替えてください。 「相手先」「共通LINE/自社LINE」の呼称は製品UIに準拠しています。

目次

0. 共通の流れ 1. LINE（共通LINE / 自社LINE） 2. Telegram 3. Chatwork 4. WhatsApp (Cloud API) 5. Slack 6. Discord（Gatewayワーカー） 7. 送信のみ（Google Chat / Teams） 8. Webhook URL 早見表 9. 環境変数チェックリスト 10. Discordワーカーのデプロイ 11. プランと上限の注意

0. 共通の流れ

どのチャネルも大きくは同じ2段構えです。

  1. TaskApp 管理画面「つなぐ」でチャネル資格情報を登録する（トークンや署名鍵を保存。値は暗号化して保管され、以後は画面に再表示されません＝ローテートで再設定）。登録すると {accountId} が発行されます。
  2. 各サービス側で Webhook / Bot を設定し、TaskApp の受信URL（早見表）を貼り付ける。以後、そのサービスの会話が AI秘書 に取り込まれ、タスク候補として承認待ちに並びます。
セキュリティ設計（共通）：受信URLはアカウントごとで、そのアカウントに保存した鍵でだけ署名検証されます。署名が合わない/鍵が無いリクエストは拒否（401）。承認前のグループの会話は1行も保存しません。Discord 以外の受信は各サービスの HTTP Webhook を使い、Discord のみ常駐ワーカー（後述）を使います。

1. LINE 受信送信

LINE には2つのつなぎ方があります。違いは「誰の名前で相手に届くか」です。

方式名義プランつなぎ方
共通LINETaskApp 共有アカウント（owner_type=platform）無料グループにコードを投入 → 管理画面で承認
自社LINE（白ラベル）事務所自身のLINE公式アカウント（owner_type=org）ProMessaging API チャネルを登録

A. 共通LINE（無料・運用者の作業は最小）

  1. 管理画面「つなぐ」→ 相手先のスペースで接続コードを発行する。
  2. そのコードを相手先に渡し、対象の LINE グループに投稿してもらう。
  3. 管理画面の「確認待ち」でグループを承認すると、以後の会話が記録されます。

※ 共通LINE の Bot 本体（Messaging API チャネル）と Webhook はプラットフォーム側で用意済みです。運用者はコード発行と承認だけ。

B. 自社LINE（Pro・自社の名前で届く）

  1. LINE Developers で Messaging API チャネルを作成。
  2. 管理画面「つなぐ」→ LINE を選び、次を登録：
    Channel secret（Webhook署名検証に使用）
    Channel access token（長期）（送信に使用）
  3. LINE 側の Webhook URL に <アプリのドメイン>/api/channels/line/webhook を設定し、Webhook 利用を ON。応答メッセージ（自動応答）は OFF 推奨。

2. Telegram 受信送信

  1. Telegram の @BotFather で Bot を作成し Bot Token を取得。
  2. 管理画面「つなぐ」→ Telegram を選び、Bot Token を登録。
    登録時に Webhook Secret Token がサーバー側で自動生成されます（次の手順で使用）。
  3. Telegram の setWebhook を呼び、URL と secret_token を設定： 
    curl "https://api.telegram.org/bot<BOT_TOKEN>/setWebhook" \
  -d "url=<アプリのドメイン>/api/channels/telegram/webhook/{accountId}" \
  -d "secret_token=<登録時に発行されたWebhook Secret Token>"

受信は X-Telegram-Bot-Api-Secret-Token ヘッダで検証されます。Bot をグループに追加し、必要なら privacy mode を無効化してください。

3. Chatwork 受信送信

  1. Chatwork で連携用アカウントの API Token を発行。
  2. 管理画面「つなぐ」→ Chatwork を選び、API Token を登録。
    登録時に /v2/me で Bot 自身の account_id を解決・検証します（自己発言のエコー防止のため。トークンが無効だと登録は失敗します）。
  3. Chatwork の Webhook（Webhook Version 2）を作成し、次を設定：
    Webhook URL <アプリのドメイン>/api/channels/chatwork/webhook/{accountId}
    イベント メッセージ作成 / 自分への返信・メンション
  4. 発行された Webhook Token を管理画面「つなぐ」の Chatwork 設定に追加登録（署名 X-ChatWorkWebhookSignature の検証に使用）。

4. WhatsApp (Cloud API) 受信送信

  1. Meta for Developers で App を作成し WhatsApp（Cloud API）を追加。System User Access Token を取得。
  2. 管理画面「つなぐ」→ WhatsApp を選び、次を登録：
    System User Access Token（Graph API 呼び出し／送信）
    App Secret（X-Hub-Signature-256 検証＝受信有効化に必要。Meta App 設定→基本設定で取得）
    登録時に Verify Token が自動生成されます（次で使用）。
  3. Meta App の WhatsApp → Configuration → Webhook で設定：
    Callback URL <アプリのドメイン>/api/channels/whatsapp/webhook/{accountId}
    Verify Token 上で発行された値
    「Verify and Save」で検証（GET ハンドシェイク）が通れば OK。messages フィールドを購読。

5. Slack 受信送信

  1. Slack API で App を作成。Bot Token（xoxb-）と Signing Secret を取得。
  2. 管理画面「つなぐ」→ Slack を選び、次を登録：
    Bot User OAuth Token xoxb-…（chat.postMessage 送信）
    Signing Secret（Events API の v0 署名検証）
  3. Slack App の Event Subscriptions を ON にし、Request URL に <アプリのドメイン>/api/channels/slack/webhook/{accountId} を設定（url_verification チャレンジが自動で通ります）。
  4. Bot Events で message.channels（必要に応じ message.groups 等）を購読し、Bot をチャンネルに招待。

署名は v0= 形式＋±5分のリプレイ窓で検証します。旧 Slack 連携とは分離した独立実装です。

6. Discord（Gatewayワーカー） 受信

Discord だけは仕組みが違います。 Discord はメッセージ受信を HTTP Webhook で取得できず、Message Content Intent（特権インテント）付きの Gateway(WebSocket) 常駐接続が必要です。そのため薄い常駐ワーカーを別プロセスとしてデプロイし、受信メッセージを AI秘書 の内部取り込み口へ HMAC 署名付きで転送します。
  1. Discord Developer Portal で Application＋Bot を作成し、Message Content Intent を ON（100サーバー未満は審査不要）。Bot Token を取得。
  2. Bot を対象サーバーに招待（View Channels + Send Messages 権限）。
  3. app 側で 共有Bot の channel_accounts(platform, channel='discord') 行をプロビジョニング（bot_token を encrypt_system_secret で暗号化保存）。手順は docs/setup/DISCORD_GATEWAY_PROVISIONING.md 参照。
  4. app 環境に DISCORD_INGEST_HMAC_SECRET、常駐ワーカーに同一値の INGEST_HMAC_SECRET を設定。
  5. 常駐ワーカー（worker/discord-gateway/）を Railway / Fly.io へデプロイ（§10）。

相手先グループの紐付け（承認/コード償還）や Pro 課金ゲートは LINE共通Bot と同じ仕組みで app 側が行います。ワーカーは「受ける・正規化する・運ぶ」だけで、DB 資格情報は持ちません。

7. 送信のみ（Google Chat / Teams / Discord Webhook）

次は送信専用（AI秘書 からの通知・返信を送るだけ）。受信（会話の取り込み）は現状ありません。

  • Google Chat 送信：スペース設定で発行した Incoming Webhook URL を登録。
  • Microsoft Teams 送信：Incoming Webhook 等の送信先を登録（許可ホストは logic.azure.com に限定）。
  • Discord（送信） 送信：各チャンネル設定→連携サービス→Webhook で発行した Channel Webhook URL を登録（受信の Gateway ワーカーとは別物）。

8. Webhook URL 早見表

各サービスの管理画面に貼り付ける受信URL（<アプリのドメイン> と {accountId} は読み替え）。

チャネル受信 Webhook URL署名/検証
LINE<アプリのドメイン>/api/channels/line/webhookChannel secret（X-Line-Signature）
Telegram<アプリのドメイン>/api/channels/telegram/webhook/{accountId}Webhook Secret Token（自動生成）
Chatwork<アプリのドメイン>/api/channels/chatwork/webhook/{accountId}Webhook Token（X-ChatWorkWebhookSignature）
WhatsApp<アプリのドメイン>/api/channels/whatsapp/webhook/{accountId}App Secret（X-Hub-Signature-256）＋Verify Token
Slack<アプリのドメイン>/api/channels/slack/webhook/{accountId}Signing Secret（v0 署名）
Discord（HTTP Webhook なし＝常駐ワーカーが /api/channels/discord/ingest へ HMAC POST）INGEST_HMAC_SECRET（HMAC-SHA256）

9. 環境変数チェックリスト

アプリ（TaskApp 本体）

  • NEXT_PUBLIC_SUPABASE_URL / NEXT_PUBLIC_SUPABASE_ANON_KEY（既存）
  • SYSTEM_ENCRYPTION_KEY — 共有Bot の bot_token を encrypt_system_secret で暗号化する際に使用（Discord プロビジョニング）
  • DISCORD_INGEST_HMAC_SECRET — Discord 常駐ワーカーからの取り込みを検証する共有鍵（ワーカーの INGEST_HMAC_SECRET と同一値）

Discord 常駐ワーカー（worker/discord-gateway/）

  • DISCORD_BOT_TOKEN — Developer Portal の Bot Token
  • INGEST_URL — <アプリのドメイン>/api/channels/discord/ingest
  • INGEST_HMAC_SECRET — app の DISCORD_INGEST_HMAC_SECRET と同一値
  • （任意）BATCH_MAX_SIZE（既定20）/ FLUSH_INTERVAL_MS（既定2000）
共有鍵は十分に長いランダム値に。例：openssl rand -hex 32

10. Discordワーカーのデプロイ

worker/discord-gateway/ に Dockerfile 同梱。単一インスタンス運用を推奨します（多重起動は同一メッセージを二重送信しますが、下流が重複を吸収するため実害はありません）。

Fly.io の例

cd worker/discord-gateway
fly launch --dockerfile Dockerfile          # min_machines_running = 1 を設定
fly secrets set DISCORD_BOT_TOKEN=... INGEST_URL=https://<アプリのドメイン>/api/channels/discord/ingest INGEST_HMAC_SECRET=...
fly deploy

Railway の例

  • このリポジトリの worker/discord-gateway を Root Directory に指定してデプロイ。
  • Variables に DISCORD_BOT_TOKEN / INGEST_URL / INGEST_HMAC_SECRET を設定。

ローカル確認：cd worker/discord-gateway && npm install && npm test（純ロジックのテスト）→ npm run build && npm start。

11. プランと上限の注意

  • LINE以外の他チャット連携（external_chat_channels）は Pro 専有。Free の相手先が Discord 等のコードを投入しても紐付けは確立しません（無効文言に畳んで情報を漏らさない設計）。
  • 外部チャットの相手先グループ数は maxExternalChatGroups（Pro=50・Enterprise=無制限・Free=0）。LINE の maxLineGroups とは別カウントで、相互に枠を汚染しません。上限到達時は新規承認のみ 402 で拒否し、既存グループは切りません。
  • 即時通知・時刻指定リマインド・自社名義LINE・担当者への個別DM も Pro 専有機能です。
価格・上限値は暫定です（LINE規約の複数アカウント可否と抽出原価の実測が前提）。maxExternalChatGroups 等はアプリ層のソフト上限で、厳密なアトミック上限が必要なら別途 DB 制約の検討余地があります（本ガイド時点では未実装）。
