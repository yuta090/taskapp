# 汎用Webhook受信 v1（送信側向け仕様）

公開APIが無い／弱いツールのタスクを TaskApp へ取り込むための受け口。**TaskApp からそのツールへ
取りに行くことはしない**。Zapier / Make / n8n などの自動化サービス、あるいは自前のスクリプトから
この仕様どおりに POST してもらう。

この文書は**外部仕様**。ここに書いた内容に顧客の送信設定が依存するため、既存フィールドの意味・
必須性・署名方式は変更しない（拡張は任意フィールドの追加のみ）。破壊的変更が要る場合は
`v2` を併設し、v1 は残す。

---

## 1. エンドポイント

```
POST https://<TaskAppのURL>/api/connectors/generic/events
Content-Type: application/json
X-AgentPM-Signature: t=<unix秒>,v1=<署名>
```

`connection_id`（受信口のID）と署名用の鍵は、TaskApp の **ツール連携 → その他のツール（Webhook）**
で受信口を作ったときに表示される。**鍵はその画面でしか表示されない**ので、その場で控えること。

受信口は既定で無効になっている。**取り込み先スペースを設定するまでイベントは受け付けない**
（設定前に大量のタスクが予期しない場所へ流れ込むのを防ぐため）。

---

## 2. 署名

```
署名対象 = "<unix秒>" + "." + <リクエストボディの生バイト列>
署名     = HMAC-SHA256(鍵, 署名対象) を小文字16進で表現したもの
ヘッダ   = X-AgentPM-Signature: t=<unix秒>,v1=<署名>
```

- **ボディは送信したそのままのバイト列**で署名する。整形し直したり、キーの順序を変えたりすると
  一致しない。
- `t` は送信時刻（UNIX秒）。**現在時刻から±5分**の範囲外は拒否される（再送攻撃を防ぐため）。
  再送するときは `t` と署名を作り直すこと。

Node.js の例:

```js
const crypto = require('crypto')
const body = JSON.stringify(payload)
const t = Math.floor(Date.now() / 1000)
const v1 = crypto.createHmac('sha256', SECRET).update(`${t}.${body}`).digest('hex')
// fetch(url, { method:'POST', body, headers:{ 'Content-Type':'application/json',
//   'X-AgentPM-Signature': `t=${t},v1=${v1}` } })
```

## 3. ペイロード

| フィールド | 型 | 必須 | 意味 |
|---|---|---|---|
| `event_id` | string(≤255) | **必須** | このイベントの一意キー。**再送時は同じ値**にする（二重取り込みを防ぐ鍵） |
| `event_type` | string | **必須** | `task.created` / `task.updated` / `task.completed` |
| `connection_id` | UUID | **必須** | 受信口のID |
| `external_id` | string(≤255) | **必須** | そのツール側のタスクID。**接続内で一意**であること |
| `title` | string(≤4000) | `task.created` のみ必須 | タスク名 |
| `body` | string(≤4000) \| null | 任意 | 本文。**`null` を送ると本文を空にする**。省略時は変更しない |
| `due_date` | `"YYYY-MM-DD"` \| null | 任意 | 期日。**日付のみ**（日時は時差で1日ずれるため受け付けない）。`null` で期日を外す。省略時は変更しない |

リクエストボディの上限は **64KB**。

```json
{
  "event_id": "andpad-9f3c-2026-07-21T10:00:00Z",
  "event_type": "task.created",
  "connection_id": "11111111-1111-4111-8111-111111111111",
  "external_id": "9f3c",
  "title": "現場写真の提出",
  "body": "A棟3F 施工後",
  "due_date": "2026-07-31"
}
```

### イベント別の挙動

- **`task.created`** — 取り込み先スペースにタスクを作る。同じ `external_id` を再度送っても
  タスクは1件のまま（作り直さない）。
- **`task.updated`** — 既に取り込んだタスクの内容を更新する。送ったフィールドだけ変わる。
- **`task.completed`** — 取り込んだタスクを完了にする。既に完了なら何も起きない。

`task.updated` / `task.completed` は、**その `external_id` が既に取り込まれている**必要がある
（先に `task.created` を送る）。

---

## 4. 応答と再送

| ステータス | 意味 | 送信側の対応 |
|---|---|---|
| `200` | 受理（または再送の重複として無視） | 何もしない |
| `400` | ペイロードが仕様に合わない（本文に理由が入る） | **設定を直す**。同じ内容で再送しても通らない |
| `401` | 署名が不正、または受信口が無効／存在しない | 鍵・`connection_id`・受信口の有効化を確認する |
| `404` | その `external_id` が取り込まれていない | 先に `task.created` を送る |
| `409` | 取り込み済みのタスクが TaskApp 側で削除されている | 新しい `external_id` で作り直す |
| `413` | ボディが大きすぎる | 64KB 未満にする |
| `422` | 受信口に取り込み先スペースが未設定 | TaskApp 側で設定する。設定後に再送すれば通る |
| `5xx` | TaskApp 側の一時的な障害 | **再送してよい**（間隔を空けて） |

**再送は安全**。同じ `event_id` のイベントは、既に処理済みなら何も起こさずに `200` を返す。
逆に、処理の途中で失敗したイベントは記録されないので、再送すれば最後まで処理される。

---

## 5. 制限と注意

- **鍵が漏れると、その受信口のスペースに誰でもタスクを作れる**。Zapier 等に登録した鍵は
  そのサービスの認証情報と同じ扱いにすること。漏れた場合は受信口を削除して作り直す。
- 期限リマインドの対象外。**この経路で取り込んだ期日は AI秘書の催促には使われない**
  （次にいつイベントが届くか TaskApp 側からは保証できず、古い期日で相手を急かす事故を避けるため）。
- 完了の書き戻しはできない（TaskApp で完了しても、そのツール側には反映されない）。
  こちらから相手を呼べないため。双方向にしたい場合は、対応済みのツール（Backlog / Jira /
  Redmine / Asana / Trello / Linear / Jooto / Google Tasks）の個別連携を使う。
