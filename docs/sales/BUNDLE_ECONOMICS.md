# バンドル経済とAI原価床（COGS）

最終更新: 2026-07-21

## 結論（前提の是正）

**agentpm秘書（TaskApp）は「ツール単体」では、営業が人手をかけて売れる単価にならない。**
月額数千〜1万円台のSaaS単価に対し、営業1商談〜導入コンシェルジュの人件費が重すぎる。
セルフサーブで無料→有料に自動転換するPLGも、活性化が人手依存で成立していない
（`docs/ops/SHARED_LINE_BUDGET_RUNBOOK.md` / 活性化ギャップ是正の経緯を参照）。

したがって売り方は次の一択：

- **上位のコンサル／multica のリテーナー（顧問料）に“機能として”束ねて売る。**
  ツールは単体SKUではなく、コンサル契約に含まれる価値の一部。営業が売るのは
  「タスク完遂の仕組み込みのコンサル」であって、ツールのライセンスではない。
  （関連: プロジェクト文脈 skara / multica ＝ Pro ハブはここに繋いで売る）

この文書は「ツールをいくらで売るか」を決めるものではない。**束ねたときに粗利が溶けないよう、
ツールが食う原価（COGS）の床を実測して見える化する**ためのもの。

## 何が原価か（2レイヤ）

| レイヤ | 原価 | 実測の仕組み |
|---|---|---|
| **LLM抽出** | プロバイダーのトークン課金（入力/出力） | `ai_usage_events`（実トークンlog）＋ `src/lib/ai/cost.ts`（単価表→円換算）|
| **共通LINE送信** | 共有アカウントのLINE無料枠(200通/月・アカウント単位)を超えた分 | `platform_channel_budget`（グローバル予算）＋ **org別cap(無料50通)は 2026-07-21 以降 `on_exceed='block'` で実際に停止** |

自社LINE（Pro）はLINE費用が事務所持ちのため、当社COGSは実質LLM分に寄る。
共通LINE（無料/共有）は当社がLINE枠と原価を負う＝**バンドル時の粗利を最も削るのは共通LINE経路**。

## 原価床の見方（実測フロー）

1. `callLlm` が呼ばれるたび、`recordAiUsage` が `ai_usage_events` に
   `org_id / provider / model / prompt_tokens / completion_tokens / purpose` を best-effort 記録
   （記録失敗は握りつぶし、抽出本体は絶対に止めない）。
2. `app_org_ai_usage_monthly` view が org×月×model のトークンを合計。
3. `src/lib/ai/cost.ts` の `MODEL_PRICES`（USD/1Mtok・代表値）と `estimateCostJpy` で円原価に換算。
   - ⚠ `MODEL_PRICES` と `DEFAULT_USD_JPY` は初期見積り値。**実トークンが溜まったら実勢価格で更新**する。
   - 未収録モデルは `null`（0円に丸めて原価を過小評価しない）。

これで「1事務所あたり月いくらのLLM原価がかかっているか」が実データで出る。

## 束ね方の指針（値付けではなく粗利ガード）

- ツールは顧問料の中の一機能。**顧問料 − (LLM原価 + 共通LINE原価 + サポート工数) が粗利**。
  CLAUDE.md の方向性（Pro は粗利70-80%・LINE費用込みで月10万円未満）を、この実測COGSで裏取りする。
- 共通LINE（無料枠）に相乗りする事務所が増えるほどグローバル枠(200/月・アカウント単位)を食い合う。
  無料の相乗り原価は非有界になり得るので、**無料はコンサル導入の入口／PoCに限定**し、
  本番の量が出る相手先は自社LINE(Pro)＝原価が事務所持ちに寄せる。
- 実測COGSが想定を超える事務所が出たら、それは「単体ツールとして薄利」ではなく
  「コンサル顧問料の設計 or Pro移行の対象」として扱う（単体値上げでは解けない）。

  `cost.ts` の単価と `DEFAULT_USD_JPY` を実勢に更新し、org別月次COGSレポートを出す。
