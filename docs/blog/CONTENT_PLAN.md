## メディアとしての設計(v0.3・「ブログ」をやめる)

- **記事の配置**: 日付ではなく体系(カリキュラム)に配置。基礎講座は順番に読める連載構造にする
  (「私も勉強したい」に応える学習導線。SEO的にもトピッククラスター構造として最適)
- **構成要素と既存計画の対応**:

  - ツール活用 = Tシリーズ(ツール別使いこなし)+C/D群
  - テンプレ集 = Excel/工程表/ガント等の配布ハブ(中間CVエンジン)
  - 用語辞典 = マイルストーン/クリティカルパス等(LLMO地層)
- **要決定(未決)**: ①メディア名(「agentpmブログ」ではない独立した名前を付けるか)
  ②URLパス(現行 /blog のまま看板だけ変えるか、/learn 等へ変えるか — 公開前の今なら変更コストゼロ)

## ポジショニング(v0.2で明確化・全記事の前提)

**agentpmは「置き換え」ではなく「アドオン」。**
すでにタスク管理・プロジェクト管理ツールを導入している(あるいは会社で決められている)のに
回っていない人・チームが主読者。もちろん単体運用もできるが、訴求の軸は
「いまの環境を捨てずに、上に足すだけで回り始める」。

この前提から導かれる読者優先順位:

1. **ツール導入済みで使いこなせていない人**(=クラスタA)← 最優先。検索者がそのまま客
2. **会社で決まったツールしか使えない人**(=E7系)← 乗り換え不要のアドオンだから拾える本命
3. **チャットに仕事が流れてきてツールに転記できていない人**(=C/D)← AI秘書の独壇場
4. **相手(顧問先・クライアント)がいて進捗が止まる人**(=B/G)← ボール概念の本命
5. これからタスク管理を始める初心者 ← 独自切り口でのみ投資(下記編集方針)

## 戦略サマリー

書く順番は **「隙間 → 計測 → 数字が出た隙間だけ深掘り」**(レビュー反映で改訂)。

1. 第1期(下記)を出し切り、12週間クラスタ別に計測する
2. 伸びたクラスタだけ増産。「網羅」は関門を通過した場合のみ解禁
3. 総合比較(H)はドメインが育つまで書かない

競合はほぼ全てツールベンダーのオウンドメディア(Backlog/Slack/Jooto/Stock/ビズクロ等)と
SaaS比較サイト(ITトレンド/アスピック)。全員「これからツールを選ぶ人」向けに書いており、
**「もうツールはあるのに回っていない人」向けの記事はほぼ空白**。ここがうちの主戦場。

## 品質ルール(レビュー反映・厳守)

- 全記事に**一次情報を最低1つ**(agentpmの実データ・顧問先の匿名実例・自社運用の失敗談)。
  AIには存在しない経験・数値・事例を生成させない(文章化支援に限定)
- 監修なしの公開はゼロ。公開ペースが監修容量を超えたら公開を止める側に倒す
- 本数はKPIにしない。KPIはクラスタ別の表示回数・中間CV・相談数

## 撤退/継続の関門

- **12週**: GSCでクラスタ別判定。伸びないクラスタは増産中止。基本系カナリア(E0/F0)が
  横ばいなら網羅展開は恒久中止
- **24週**: ブログ起点のagentpm登録が累計10件未満かつmultica問い合わせ0件なら新規執筆を
  停止し、営業同伴転用とhow-to保守に縮退。1件でも成約した型は増産

### 編集方針(ユーザー意向・厳守)

- **初歩の初歩から書く。ただし書き方を2種類に分ける**(v0.2):
  - 投資しない: 定義・メリデメ等の「どこにでもあること」はAI検索に要約されて終わる。
    内部リンク用の短い用語解説にとどめる
  - 投資する: 立場別・つまずき別・「できない人の味方」路線。ネット上に元ネタがない
    独自切り口は、AI検索が答えを作るときの引用元(LLMO)になれる
- **人力の管理法も対等に扱う**: 紙・手帳・Excel・ホワイトボードを「ツールの前座」ではなく
  正式な選択肢として書く(信頼獲得と検索網羅の両方に効く)
- **会社の制約がある読者を見捨てない**: 「決まったツールしか使えない」環境での工夫は
  アドオン・ポジショニングの本命読者。E7系は削らず主力に置く
- **いきなり記事は書かない**: カテゴリ→記事ツリー→見出し構成→執筆の順。
  見出し設計の段階でOpenSEO(契約済み)の検索データを使う

### 発見: アドオン戦略の本当の入口

「定着しない」と検索する人はいないが、**「notion タスク管習」「slack タスク管理」「backlog 使い方」と
検索する人が大量にいる** — これが「ツール導入済みで回っていない人」の実際の検索行動。
アドオン戦略は正しかったが、入口の語は「[ツール名]×タスク管理/使い方」だった。
→ 新シリーズ **「T: ツール別使いこなし」** を新設(Notion/Slack/Backlog/Asana/Excel…)。
実装済み連携9種と完全に噛み合い、記事の結論が「そのツールのまま、agentpmを足す」になる。

## Uシリーズ: ユースケース・インタビュー(事例交換プログラム・2026-07-24ユーザー発案)

**「日本中のタスク管理ユースケースを集めたサイト」を実体化する仕組み。営業と一体で回す。**

### 仕組み(事例交換プログラム)

- 新規顧客へのアプローチ時に「**ブログに掲載してよい具体的なエピソード(社名は匿名)を
  提供してくれたら、初期導入費用を割引**」というオファーを出す
- インタビュー自体が商談を兼ねる(相手の業務の困りごとを深く聞く=最高の営業ヒアリング)。
  「インタビューとして営業する」記事体系
- 集めたエピソードは EVIDENCE_LOG に外部提供分として登録(提供元・許諾条件・匿名化度合いを記録)し、
  ①Uシリーズ記事(その会社のユースケース紹介) ②業種別記事の実話 ③基礎Q&Aの悩み例、の3用途で使い回す

### 記事の型(1話完結・量産型)

1. 業種と組織規模(匿名。例:「従業員15名の建設会社」)
2. 何に困っていたか(エピソード本体。感情と具体性を残す)
3. 何を試して、どうだったか(人力の工夫も含めて正直に)
4. いまどうしているか(agentpm導入企業なら before/after。未導入でも成立させる)
5. 同じ業種の読者への示唆

### 効果(三方一両得)

- **営業**: 割引オファーで新規接触の口実ができ、インタビューで関係が深まる
- **コンテンツ**: 一次情報の在庫が代表の記憶に依存せずスケールする(E-E-A-T問題の構造的解決)
- **顧客**: 初期費用が下がり、自社の課題が言語化される(インタビュー自体に価値)

### 注意(厳守)

- **誇張・美化しない**: エピソードは事実のまま。成果の数字を盛らない(体験談の水増しは
  ステマ・優良誤認のリスク)。割引と引き換えに「良い感想」を求めるのではなく、
  **「本当にあった話」を求める**(失敗談こそ歓迎、と明示する)
- **開示**: 事例提供に対して割引等の対価がある場合は、記事側にその旨をさりげなく明示するのが安全
  (例:「本記事は事例提供プログラムにご協力いただいた企業への取材に基づきます」)
- **匿名化と許諾**: 掲載前に本人確認(業種・規模の表記レベルまで合意)。EVIDENCE_LOGに許諾を記録
- 割引の設計(額・条件)は価格戦略側で別途決める(ブログ計画の範囲外)
