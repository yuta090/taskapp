# 一次情報台帳(EVIDENCE_LOG)

ブログ記事に使う一次情報(実例・実データ・失敗談)の在庫。
品質ルール「全記事に一次情報を最低1つ」の原資であり、AI生成の一般論との唯一の差別化。

- 記事で使う際は本台帳のIDを記事管理メモに紐づける
- 匿名化の度合いは各エントリの「公開許容」に従う(未確認のものは使用前に確認)
- AIに実例を創作させることは禁止。ここにあるものだけを使う

- **記録日**: 2026-07-24
- **内容**:
  顧問先の開発を請け負い、デザインのやり取りをしていた際、「いま誰の確認待ちなんだっけ?」

  レビューが遅れ、納期に遅延が出そうになった。
  もっと確認(催促)すればよかったが、**先方が高圧的でスタッフが萎縮してしまい、催促できなかった**。
- **この話が証明すること**:

  2. 催促は「やればいい」ものではない。**力関係で人が催促できない場面が現実にある**
  3. だから催促は人ではなく仕組みにやらせる(誰も悪者にならない)— agentpmの核心の由来
- **使える記事**: G3(ボールはどっちにある?)★本命 / B2(催促せずに資料回収) /
  B3(先方待ちの見える化) / G2(停滞案件の動かし方) / G4(高圧的な相手との仕事の進め方)

## E-002: 「上司の管理のため」に導入されたNotionがメモ帳と化し、本人が通院に至った件

- **種別**: 実例(身近な現場の観察)
- **記録日**: 2026-07-24
- **内容**:
  Notionでのタスク管理を導入したが、導入の動機が「上司が部下のタスクを管理したい」であり、
  使う本人はタスク管理ソフトの有効な使い方を身につけられないまま、Notionは単なるメモ帳と化した。
  結果、その人は思うようにタスク管理ができず、精神的に不安定になり通院することになった。
- **この話が証明すること**:
  1. **導入の目的が「管理する側の都合」だと、使う本人に価値が生まれず定着しない**
     (ツール選定の問題ではなく、目的設定の問題。Notionは悪くない)
  2. 「なぜタスク管理が必要か」を本人が腹落ちしないままツールだけ渡しても、機能しない
     — ツールの前に考え方、という本ブログの編集方針の実証
  3. タスク管理の失敗は業務効率の問題にとどまらず、**人の心身を壊しうる**。
     「できない人の味方」路線は道徳ではなく実害への処方箋
- **使える記事**: E1(なぜタスク管理が必要か)★本命 / A1(定着しない理由) /
  A3(運用ルールの作り方=誰のための管理か) / E3(立場別・上司は部下の何を管理すべきか) /
  E13(苦手な人の克服法)
- **執筆時の注意**: 通院・メンタル不調の描写は本人が特定されない抽象度で、かつ扇情的に
  使わない(「体調を崩すところまで追い込まれる」程度の表現に抑える判断もあり)。
  Notionという固有名は出さず「あるタスク管理ツール」とするか要判断(ツール批判ではなく
  導入目的の問題として書くなら固有名は不要)
- **公開許容**: **匿名化前提で公開OK**(2026-07-24 本人確認済。通院描写の抑制方針は執筆時注意に従う)

## E-003: 分解できない大粒度タスクに「Xさんに確認」が隠れていて間に合わなかった件

- **種別**: 実例(身近な現場の観察)
- **記録日**: 2026-07-24
- **内容**:
  タスクの粒度が大きすぎるのを分解するのが苦手な人がいて、いつも期日前ギリギリに
  タスクを実行していた。そのタスクの中には「Xさんに確認」という工程が含まれていたが、
  実行した当日Xさんは休みだったため、タスクが間に合わなかった。
  タスクの分解を(multicaで)早めにやっていれば防げた。
  しかし現実には、タスクを1日1時間未満の粒度に分解して手動で登録するのは非常に手間が
  かかるため、**手動での分解は定着しない**。
- **この話が証明すること**:
  1. 大粒度タスクの本当の危険は作業量の見誤りではなく、**中に「他人への依存(確認待ち=ボール)」が
     隠れていること**。ギリギリ実行だと依存が発覚した時点で手遅れ(相手の不在は自分では制御できない)
  2. 「分解しましょう」という正論は皆知っている。**定着しないのは意志ではなく登録コストの問題**
     — だから分解はAIにやらせる、が処方箋になる(precept→製品の必然性)
  3. 期日ギリギリ実行型の人へのアドバイスは「早くやれ」ではなく「早く分解だけしろ(実行は後でいい)」
- **使える記事**: E4(タスクの粒度の決め方)★本命 / E9(やり方6ステップ・分解) /
  G3(ボール=隠れた確認待ち) / A6(入力・登録の手間問題) / D3(AIによる自動分解) / F2(WBS)
- **執筆時の注意**: 「Xさん」は役割名(例: 決裁者・レビュー担当)に置き換える
- **公開許容**: 匿名化前提で公開OK(2026-07-24 包括確認済)

     持っていく仕組みがないと実行されない — E-003(手動登録の手間)と対をなす、

  D系(AIをタスク管理に使う正しい方法・ChatGPTの限界) / G2(停滞の原因=分解不能) /
  E-できない人の味方系(抱え込み)
- **記事化の切り口メモ**: 「KPI・目標をタスクに落とす手順」はSEO語としても検証価値あり
  (目標管理/OKR/KPI 達成 方法 など。次回の実数調査に含める)

## E-005: 毎月のタスクを何ヶ月分も先作りして、完遂率が崩壊した件

- **種別**: 実例(身近な現場の観察)
- **記録日**: 2026-07-24
- **内容**:
  毎月実行する定例タスクを、事前に何ヶ月分もまとめて作っている人がいた。
  その結果、大量のタスクが未完了のまま並ぶ状態が常態化し、「タスクが完了しないのが当たり前」の
  感覚が生まれ、タスクの完遂率が著しく下がった。
- **この話が証明すること**:
  1. 未完了タスクの山は単なる「散らかり」ではなく、**「終わらなくて当たり前」という感覚を育てて
     完遂率そのものを下げる**(割れ窓理論のタスク版。通知の来すぎで通知を見なくなるのと同じ構造)
  2. 繰り返しタスクの設計問題: 大事なのは「作っておくこと」ではなく**「発生させるタイミング」**。
     直前に自動生成されるべきで、人間が先作りした瞬間にリストは在庫の山になる
  3. **タスクリストは信頼性が命**。死んだタスクが混ざるとリスト全体が信用されなくなり、
     結局リストの外(記憶・チャット)で仕事が回り始める — ツールがメモ帳化する(E-002)前段階
- **使える記事**: A3(運用ルール=リスト衛生)★本命 / B4(会計事務所の月次業務=定例タスクの
  正しい複製運用) / Tシリーズ(各ツールの繰り返しタスク機能の正しい使い方) /
  E11(抜け漏れ=信頼できるリストの条件) / G5(完了確認)
- **記事化の切り口メモ**: 「タスク 未完了 溜まる」「todoリスト 消化できない」系の語は次回実数調査へ
- **公開許容**: 匿名化前提で公開OK(2026-07-24 包括確認済)

## E-006: グループチャットで依頼した評価面談のスケジュール登録がリマインドされず、従業員評価が未実施になった件

- **種別**: 実例(身近な現場の観察)
- **記録日**: 2026-07-24
- **内容**:
  上司がスタッフとの**評価面談**を実施する必要があり、面談スケジュールの登録をチャットグループで
  スタッフに依頼した。しかし「まだ登録していないスタッフ」に対するリマインドがされず、
  一部スタッフの評価面談が行われないまま、従業員評価が未実施となり大問題になった。
- **この話が証明すること**:
  1. **グループへの一斉依頼は「全員のタスク」になり、誰のタスクにもならない**(責任の拡散)。

  3. タスク管理の失敗の帰結は業務遅延にとどまらず、**評価・人事という組織の信頼問題に直結**する
     (E-002の心身、E-006の人事 — 「たかがタスク管理」ではない実害の系譜)
- **使える記事**: C2(グループの依頼を拾い漏らさない)★本命 / G5(完了確認=「やったつもり」を
  なくす) / E-001と対で「催促は仕組みにやらせる」系 / E3(立場別=管理職の依頼の出し方) /
  B系(スタッフ・チーム運営)
- **製品接続**: 秘書のグループ拾い+期限リマインド+完了確認ループがこの構造の直接の処方箋
  (機能紹介ではなく「未完了者だけを機械が追う」という設計思想として書く)

     変わり、誰も悪者にならない — 送る側のストレスも受ける側の反感も同時に消える
  3. 経理・総務などバックオフィスは「社内全員に催促する立場」であり、

  - E-007: 催促が**ストレス**(嫌われ役・経理側)
  → 三方向から「催促は人がやってはいけない。仕組みにやらせろ」というピラー主張が立つ

## E-008: 納品後の運用でバグ報告と追加要望が混在し、GitHub Issuesは非技術者に使えなかった件

- **種別**: 実例(受託開発の現場)
- **記録日**: 2026-07-24
- **内容**:
  プログラム開発で納品完了後の運用フェーズに入ると、**バグの報告と新規機能の追加要望が
  混ざってしまい**、管理が煩雑になった。当初はGitHubのIssueで管理していたが、
  **UIが技術者でない人間には使いづらく**、定着しなかった。
- **この話が証明すること**:
  1. 納品後の運用は「バグ(直す義務)」と「追加要望(別料金・要判断)」の**性質が違うタスクが

     開発者標準(GitHub Issues)は、報告する側=客・非技術者には高すぎる壁だった
  3. 客と作り手が**同じ場所で・それぞれに合った見え方で**タスクを共有できる仕組みの不在が根本原因

  使いこなしと限界) / E6(ツールの種類=開発者向けと非技術者向けの断層) /
  E-001と同じ受託文脈(顧問先との共有) / バグと要望の仕分け方(運用保守の型)

- **種別**: 実例(受託開発の現場)
- **記録日**: 2026-07-24
- **内容**:

- **この話が証明すること**:

  3. E-001(誰の確認待ちか不明)の変種: こちらは「**何が未決かすら管理されていない**」パターン。

  agentpmの会議まわり機能群の思想的裏付け
